Observability August 2024

Stop logging strings. Start logging structured events.

This is a log message: User 4821 checked out cart with 3 items totalling $47.20.

This is a structured event: {"event": "checkout.completed", "user_id": 4821, "item_count": 3, "total_cents": 4720, "currency": "USD"}.

The string is readable. The structured event is queryable. That difference compounds fast.

The cost of unstructured logs at scale

When your service handles a thousand requests a second, grep is not an observability strategy. Unstructured log messages require regex parsing to extract values, which is brittle, slow, and has to be redone every time the message format changes. The person who changes a log message rarely knows what dashboards or alerts depend on parsing it.

Structured logs emit data as key-value pairs. Your logging infrastructure can index on those keys. You can filter on user_id = 4821 without a regex. You can aggregate on total_cents without a pipeline transform. You can add a new field without breaking existing queries.

What makes a good structured event

A good event has a stable event field that names what happened. All varying data lives in typed fields. Timestamps are ISO 8601. IDs are strings, not integers. Boolean states are explicit fields, not embedded in the event name ("checkout.completed" with a success: false field, not "checkout.failed" as a separate event type).

Start small

You don't need to retrofit your entire codebase. Pick one high-value flow — the payment path, the auth layer, the job queue — and instrument it with structured events. Build the habit and the infrastructure around that, then expand. Within a few months, you'll find yourself reaching for your structured logs first and wondering why you ever logged strings.
