A consensus algorithms primer for engineers who build on top of them

You don't need to implement Raft to use etcd or ZooKeeper well. But you do need a working mental model of what consensus algorithms guarantee — and what they don't — to avoid the class of bugs that come from misusing them.

What consensus actually guarantees

A consensus system like Raft guarantees that a cluster of nodes will agree on a sequence of values, even if some nodes fail, as long as a majority remain available. The key properties are safety (all nodes that decide, decide the same value) and liveness (the cluster eventually makes progress if enough nodes are up).

What it does not guarantee: performance under adversarial network conditions, sub-millisecond latency, or progress during a network partition. The CAP theorem is real — consensus systems choose consistency over availability during a partition.

Leader election is not instantaneous

When a Raft leader fails, the cluster must elect a new one. This takes time — typically 150–300ms in a well-tuned cluster. During this window, the cluster is unavailable for writes. If your application assumes the leader is always available, you'll see errors during elections. Design for this: use retries with backoff, and distinguish between "leader unavailable" and "operation failed".

Read consistency is a choice

Reading from a Raft follower is fast but may return stale data. Reading from the leader is linearisable but adds latency and load. Most systems expose this as a configuration option. Know which one your workload requires — many applications that think they need linearisable reads actually only need sequential consistency, which followers can provide with a small protocol extension.
