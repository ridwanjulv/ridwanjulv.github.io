Build systems at scale: what breaks and why

At some point in every growing engineering organisation, the build stops being fast. This happens gradually, then suddenly. A CI run that took 4 minutes now takes 40. Nobody changed anything dramatic — it's the compounding cost of a hundred small decisions made when the codebase was smaller.

The cache invalidation problem is usually the build problem

Most slow builds are slow because of cache misses, not because of actual work. A test suite that re-runs 800 tests when you change a utility function has a dependency graph problem. Bazel, Buck, and Pants exist to solve this: they model your build as a directed acyclic graph and only rebuild what actually changed.

The upfront cost is real — these tools require explicit dependency declarations and a shift in how you think about code organisation. The payoff at scale is equally real: builds that are O(change size) rather than O(codebase size).

Test parallelism has diminishing returns

Throwing more cores at a slow test suite helps, up to a point. The limit is usually I/O — database connections, file handles, network calls in tests that shouldn't make them. Profiling your test suite before scaling your CI infrastructure often reveals that 20% of tests account for 80% of runtime, and most of those have fixable problems.

Flaky tests are a build system problem

A test that fails 5% of the time is worse than a test that always fails, because it's invisible. Engineers learn to re-run CI until it passes. Flakiness is technical debt with interest. Track flakiness rates explicitly, assign ownership, and treat flaky tests as P2 bugs. The teams that do this have fundamentally different CI cultures than the teams that don't.
