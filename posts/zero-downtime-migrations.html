Postgres February 2024

Zero-downtime migrations are possible. Here's the checklist.

Most database migrations don't need downtime. The ones that cause incidents usually do so because of a lock taken at the wrong time, not because the change itself required taking the database offline.

The checklist

Adding a column: Safe if nullable or has a server-side default. Dangerous if you add a non-nullable column with no default on a large table — Postgres will rewrite the table. Use a nullable column, backfill, then add the constraint.

Dropping a column: Safe in two steps. First deploy code that stops reading/writing the column. Then drop it. Dropping first causes errors in any running version of your code that still references it.

Adding an index: Always use CREATE INDEX CONCURRENTLY. The non-concurrent form takes a write lock for the entire build duration. On a 50GB table, that's a long maintenance window.

Renaming a column: Never do it in one step. Add the new column, dual-write, backfill, migrate reads, remove the old column. Four deploys minimum.

Changing a column type: Requires a table rewrite in most cases. Use a shadow column, backfill, swap at the application layer, clean up. Same pattern as rename.

The rule underneath the checklist

Every migration that touches existing data needs to be compatible with both the old and new version of your application running simultaneously. During a rolling deploy, you'll have both versions live. Your schema change has to work for both. This is the constraint that generates all the multi-step patterns above.

Build this constraint into your review process and most migration incidents disappear.
